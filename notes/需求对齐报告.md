## 一、项目现状分析

### 1.1 代码审计结果

#### 🔴 严重问题（阻断性）

| # | 文件 | 问题 | 影响 |
|---|---|---|---|
| S1 | `harvest_api.py` | `from app.core.harvest.smart_scheduler import DatabaseManager, init_tables, save_harvest_records` — `SmartScheduler` 模块中并不存在这些导入 | API 无法启动 |
| S2 | `harvest_api.py` | `DatabaseManager` 从 `smart_scheduler` 导入，但实际定义在 `async_models.py` | 运行时 `ImportError` |
| S3 | `harvest_main.py` | `HarvestRecord(...)` 使用 `source=`、`title=`、`url=` 等参数，但 `async_models.py` 中 `HarvestRecord.__init__` 的第一个参数是 `self`，无 `source` 字段（实际字段是 `source_name`） | CLI 运行时崩溃 |
| S4 | `harvest_main.py` | `from scripts.async_chongqing_procurement_refactored` — 该脚本可能不存在 | CLI 启动失败 |
| S5 | `harvest_main.py` | `CacheManager` 在 `_save_results` 中使用 `CacheManager._make_key()` 但 `_make_key` 是静态方法，与 `RedisManager` 混用 | 缓存功能失效 |

#### 🟡 中等问题

| # | 文件 | 问题 | 影响 |
|---|---|---|---|
| M1 | `harvest_api.py` | 内部类 `SiteCrawler(DefaultCrawler)` 定义在 `trigger_crawl` 函数内部，每次请求都重新创建类 | 内存泄漏风险 |
| M2 | `harvest_main.py` | `_build_scheduler` 返回 tuple `(priority_engine, interval_manager)` 但函数签名无类型标注 | 调用方困惑 |
| M3 | `async_models.py` | `HarvestRecord.create()` 方法直接 `raise NotImplementedError` 后调用 `cls.__name__`（会返回字符串 "HarvestRecord" 而非实例） | 死代码 |
| M4 | `async_models.py` | `INIT_TABLES_SQL` 中 `CREATE INDEX IF NOT EXISTS` 与 `CREATE TABLE IF NOT EXISTS` 在同一 `DO $$` 块中，先天不足 | PostgreSQL 方言问题 |
| M5 | `smart_scheduler.py` | `SmartScheduler` 的 `schedule()` 方法缺少与 `ExceptionStateMachine` 的集成 | 异常时无法自动恢复 |
| M6 | 测试文件 | `test_integration.py` 中 `from async_crawler_base import AsyncCrawlerBase` — 路径为 `scripts/async_crawler_base`，但合并后已迁移到 `app/crawlers/async_base/__init__.py` | 测试无法运行 |

### 1.3 各专家评审意见

#### 后端架构师
- 模块边界不清晰：`harvest_api.py` 应只依赖 `SmartScheduler`，不应该直接导入数据库层
- 建议：引入 Repository 模式，让 `SmartScheduler` 通过 Repository 与数据库交互
- 建议：`harvest_api.py` 中的爬虫工厂应提升为模块级单例

#### AI工程师
- `DynamicPriorityEngine` 目前只支持 5 个因子，缺少站点类型（公告板/中标/采购）差异化支持
- `DemandTracker` 目前是内存存储，生产环境应接入 Redis
- 建议：引入基于历史的成功率预测（简单线性回归或梯度下降）

#### DevOps工程师
- **缺失**：Dockerfile、docker-compose.yml
- **缺失**：环境变量配置示例 (`.env.example`)
- **缺失**：health check 端点可增强（应包含 Redis/PostgreSQL 探活）
- 建议：使用 `gunicorn` + `uvicorn` 多进程部署 FastAPI

#### 安全工程师
- `security.py` 中的 `RateLimiter` 是内存实现，分布式部署下无效
- `CSRF_HEADERS` 常量定义但未使用
- API 的 `/crawl` 端点无认证，任何人可触发采集任务
- 建议：引入 API Key 认证或 JWT

#### 代码审查员
- `exception_handler.py` 与 `smart_scheduler.py` 完全独立，未集成
- `AnomalyClassifier` 的关键词列表有硬编码中文（如 "请验证"），但错误消息未必是中文
- 测试覆盖不足：关键路径（API 端点、调度器、异常恢复）缺少单元测试
- 代码风格：`exception_handler.py` 中有大量 `logger.warning/error` 但没有结构化日志（JSON）
