## 二、架构优化方案

### 2.1 核心改进

| 改进点 | 现状 | 优化后 | 收益 |
|--------|------|--------|------|
| **异步采集** | requests (同步) | aiohttp (异步) | 并发 20x |
| **连接池** | 每次新建 | 复用 + DNS缓存 | TCP开销 -90% |
| **限流** | 固定 sleep | 令牌桶算法 | 精确控制 |
| **重试** | 盲目重试 | 智能分类 + 退避 | 封禁率 -80% |
| **存储** | SQLite | PostgreSQL | 查询速度 25x |
| **缓存** | 无 | Redis 多级缓存 | 命中率 40% |

## 五、AI 智能化方案

### 5.1 核心改进

| 改进点 | 现状 | 优化后 | 收益 |
|--------|------|--------|------|
| **调度** | 静态优先级 | 动态优先级 | 关键任务及时 +40% |
| **间隔** | 固定 sleep | 自适应 | 封禁率 -60% |
| **异常** | 被动重试 | 智能分类自愈 | 自愈率 >80% |
| **质量** | 无评估 | 四维评分 | 质量 >90分 |
| **扩容** | 人工 | 预测性扩容 | 资源浪费 -30% |
| **查询** | 无 | 自然语言 | 用户体验 +∞ |

### 5.2 智能调度公式

```
优先级_score = w1×时效性 + w2×可靠性 + w3×成本 + w4×成功率 + w5×需求强度

权重配置：
w1=0.25 (时效性) - 截止前快速上升
w2=0.20 (可靠性) - 来源可用性滑动均值
w3=0.15 (成本)   - 归一化采集成本
w4=0.20 (成功率) - 近7天成功率
w5=0.20 (需求)   - 查询频率/订阅数
```

### 5.3 异常分类与响应

| 异常类型 | 信号 | 自动恢复策略 |
|----------|------|-------------|
| rate_limit | 429 / retry_after | 长退避 5-15min |
| ban | 403 / captcha / blocked | 切换代理/站点 |
| network_timeout | timeout / reset | 短重试 10-30s |
| parse_error | selector / null | 切换备选选择器 |
| server_error | 500/502/503 | 等待恢复后重试 |

## 十、结论

本次优化方案覆盖了采集系统的**全链路改进**，从基础设施（Docker、监控）到核心业务（异步采集、智能调度），从安全底线（密钥管理、输入验证）到长期价值（AI 智能化）。

**核心建议：**
1. **立即行动（P0）**：安全修复不等待，密钥管理和输入验证先行
2. **稳步推进（P1）**：DevOps 基础和架构升级并行，2 周内完成
3. **持续优化（P2+）**：AI 智能化按需实施，不追求一步到位

优化后的系统将具备：**高并发、高可用、高安全、可观测、智能化** 五大特性，为业务增长提供坚实的技术支撑。
