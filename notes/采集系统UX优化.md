# 采集系统 UX 优化方案

> 角色：UX 研究员
> 日期：2026-04-07
> 依据：代码审查（scripts/ + templates/dashboard_v2.html）

---

## 一、用户旅程分析与痛点

### 1.1 完整旅程拆解

```
[创建任务] → [配置参数] → [启动执行] → [监控状态] → [查看结果] → [导出数据]
```

| 阶段 | 当前实现 | 用户动作 |
|------|---------|---------|
| 创建任务 | 手动编辑 Python 脚本 | 修改 `KEYWORDS`、`BASE_URL` 常量 |
| 配置参数 | 同上 + db_models SourceConfig | 直接 SQL 或代码修改 |
| 启动执行 | 命令行 `python script.py` | 手动终端操作 |
| 监控状态 | 无实时监控 | 等待脚本完成后看日志 |
| 查看结果 | Excel 文件 + Dashboard | 手动打开文件或刷新页面 |
| 导出数据 | 脚本自动生成 Excel | 找到文件路径，手动下载 |

### 1.2 核心痛点识别

#### P0（阻塞级）
1. **任务创建无界面** — 用户必须懂 Python 才能添加关键词/来源，门槛极高
2. **执行状态不可见** — 运行中/成功/失败/重试状态完全不透明，用户只能等
3. **异常处理无感知** — 429限速、IP封禁、解析失败全部在代码里静默处理，用户不知道采集已中断

#### P1（严重级）
4. **配置分散** — 关键词、站点URL、限速参数分散在多个文件（脚本 + db + env）
5. **结果需手动查找** — Excel文件生成在日志目录，无统一出口
6. **Dashboard 无实时更新** — 页面靠手工刷新，数据 stale

#### P2（不便级）
7. **筛选维度单一** — 当前仅支持"业务类型"筛选，无时间/关键词/来源筛选
8. **批量操作缺失** — 无法批量导出、批量标记、批量删除
9. **没有任务历史** — 一次运行对应一个 Excel，无法对比不同时间段

## 二、信息架构优化

### 2.1 导航结构设计

```
采集系统 (顶栏导航)
├── 📊 Dashboard      ← 首页，综合看板
├── 📋 任务管理       ← 任务列表 + 状态
├── ⚙️  采集配置      ← 来源配置 + 关键词管理
├── 📁 结果中心       ← 历史数据 + 导出
├── 📈 站点健康       ← 各站点成功率/响应时间
└── 🔔 异常日志       ← 告警历史 + 恢复记录
```

**关键优化**：
- 将"任务管理"独立出来，不再隐藏在脚本执行中
- "站点健康"承载 SmartScheduler 的 Metrics 数据，让黑盒变白盒
- "异常日志"承接 ExceptionStateMachine 的状态历史

### 2.2 页面层级规划

```
第一层: Dashboard（默认着陆页）
  ↓ 点击"任务管理"
第二层: 任务列表（Tab: 运行中 / 已完成 / 已失败）
  ↓ 点击任务
第三层: 任务详情（实时日志 + 中途异常 + 结果预览）
  ↓ 点击"新建任务"
第三层: 任务创建向导（Step-by-step）

第一层: 采集配置
  ↓ 点击来源
第二层: 来源详情（URL + 限速 + 选择器 + 备选方案）
```

### 2.3 搜索与筛选设计

| 维度 | 建议 |
|------|------|
| 关键词 | 模糊匹配标题 + 标签chips |
| 来源站点 | 多选下拉 + 支持站点别名 |
| 时间范围 | 快捷选项（今日/本周/本月）+ 自定义 |
| 业务类型 | 政府采购 / 工程建设（已有）|
| 采集状态 | pending / running / succeeded / failed（新增） |
| 金额范围 | 预算金额滑块 |

## 三、交互设计建议

### 3.1 采集配置向导（最高优先级）

**背景**：当前创建采集任务 = 写代码，极度不友好。

**设计方案：3-Step 向导**

```
Step 1: 选择来源
  ├── 下拉选择已有站点（ccgp-chongqing / cqggzy 等）
  ├── 或"新增站点"（输入名称 + URL）
  └── 提供 URL 健康检查（点击"检测"按钮，实时返回是否能访问）

Step 2: 配置采集参数
  ├── 关键词 chips 输入（+ 可选"排除词"）
  ├── 采集类型多选（招标公告 / 中标结果 / 采购意向）
  ├── 预算下限（金额范围筛选，可选）
  └── 定时计划（单次 / 每天 / 每周）
  └── 高级：
      ├── 最大并发数（滑块，默认 5）
      ├── 请求间隔（默认 2s，支持自动自适应）
      └── 超时时间（默认 30s）

Step 3: 预览确认
  ├── 汇总配置摘要
  ├── 预估采集量提示（基于历史数据）
  └── 预计耗时估算
```

### 3.2 实时状态展示

**背景**：当前脚本执行时用户不知道内部发生了什么。

**设计方案：在 Dashboard 增加实时状态栏**

```
顶部状态条（始终可见）:
[🔵 运行中] 政府采购  进度 67/200  │  异常 2（已自动恢复）  │  耗时 00:03:24

  ↓ 点击展开
  └── 实时日志面板（可折叠）
      ├── [19:53:01]  GET ccgp-chongqing.gov.cn  → 200 OK  (312ms)
      ├── [19:53:02]  ⚠️ 429 Rate Limit，启用退避（5s）
      ├── [19:53:08]  RETRY  → 200 OK  (289ms)
      └── [19:53:10]  📥 已采集 68/200 条（匹配 12 条关键词）
```

**技术实现**：
- 任务执行时开启 WebSocket 或 SSE，推送实时日志
- 前端 WebSocket 订阅 `/api/ws/tasks/{task_id}/logs`
- 后端 `smart_scheduler.py` 中增加 `on_progress callback`

**异常恢复可视化**：
- 当 ExceptionStateMachine 触发 SWITCH_PROXY 或 BACKOFF_LONG 时：
  - 卡片级提示：`⚠️ 站点限速，自动切换备选选择器`
  - 气泡通知（不打断操作，可忽略）
  - 异常日志自动记录，供事后复盘

### 3.3 异常处理界面

**背景**：ExceptionStateMachine 是纯代码实现，用户完全无感知。

**设计方案：独立"异常日志"页面**

```
🔔 异常日志
├── 筛选：全部 / 已恢复 / 需关注 / 已放弃
├── 时间：最近1小时 / 今天 / 近7天

每条异常卡片：
┌─────────────────────────────────────────────────────┐
│ 🔴 RATE_LIMIT          2026-04-07 19:53:02          │
│ 站点：ccgp-chongqing.gov.cn                          │
│ 详情：HTTP 429 - too many requests                   │
│ 动作：已自动退避 300s                                │
│ 结果：✅ 恢复成功（第 1 次自动恢复）                  │
│ 关联任务：task_20260407_001                          │
│                                                    │
│         [查看任务]  [查看站点健康]  [忽略]           │
└─────────────────────────────────────────────────────┘
```

**分级告警**：
- `low`（网络超时）：静默自动恢复，不主动通知
- `medium`（解析失败、Server Error）：记录 + toast 提示
- `critical`（IP封禁、连续失败）：页面级告警 + 推送通知

## 四、其余优化建议

### 4.1 结果中心设计

- **统一入口**：所有历史 Excel 在"结果中心"列出，不再需要找文件路径
- **批量导出**：勾选多条记录 → 导出 Excel / JSON / CSV
- **数据预览**：点击记录可直接在页面内预览详情（弹窗），无需打开 Excel
- **对比视图**：选择两个时间段的数据，可视化对比差异（新出现/消失的关键词）

### 4.2 站点健康看板

当前 SmartScheduler 已有的 metrics 数据（成功率、平均响应时间、被禁状态）完全未展示给用户。

```
站点健康
├── 实时状态指标：
│   ├── 成功率（环形图）
│   ├── 平均响应时间（折线图，近7天趋势）
│   ├── 当前限速状态（正常 / 限速中 / 被禁）
│   └── 调度优先级（1-10 分数）
├── 异常类型分布：
│   └── 饼图（rate_limit / parse_error / server_error / ban）
└── 推荐操作：
    └── "当前 5 个站点均稳定" / "⚠️ ccgp-chongqing 被限速，建议降低并发"
```

### 4.3 关键词管理独立页面

当前关键词硬编码在脚本中：
```python
KEYWORDS = ["智能化", "音视频", "AI", ...]
```

**优化**：独立管理界面：
- 关键词库（增删改查，支持分组/标签）
- 命中统计（每个关键词的累计命中次数）
- 推荐关键词（基于已有数据中高频出现的词根扩展）

### 4.4 定时任务集成

- 当前定时靠 cron 或手动触发
- 建议在系统内建"定时任务管理"（展示下次执行时间/最近执行结果）
- 可直接启用/停用，无需改脚本

## 五、技术落地优先级建议

| 优先级 | 功能模块 | 工作量估算 | 价值 |
|--------|---------|-----------|------|
| P0 | 任务创建向导 | 中 | 解锁核心可用性 |
| P0 | 实时状态栏 + WebSocket 日志 | 中 | 消除黑盒焦虑 |
| P0 | 异常日志页面 | 小 | 告警可视化 |
| P1 | 结果中心（统一出口） | 小 | 减少用户找文件成本 |
| P1 | 站点健康看板 | 小 | 利用已有 metrics 数据 |
| P1 | Dashboard 多维筛选 | 中 | 提升数据探索效率 |
| P2 | 关键词管理独立页面 | 中 | 配置集中化 |
| P2 | 定时任务管理 | 中 | 减少人工操作 |
| P2 | 数据对比视图 | 中 | 差异化功能 |
