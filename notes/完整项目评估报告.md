### 1.2 模块划分评估

| 目录 | 评估 | 说明 |
|------|------|------|
| `app/api/routes/` | ⚠️ **过度拆分** | 16 个路由文件，部分职责不清。如 `pages.py`（分页？）和 `presets.py`（预设配置）职责重叠 |
| `app/core/harvest/` | ⚠️ **膨胀严重** | 6 个模块合计 ~4K LOC，anti_detect.py 单独 1476 LOC，功能边界模糊 |
| `app/database/` | ❌ **架构冲突** | 同时存在 `db.py`（同步 SQLite）和 `async_models.py`（异步 PostgreSQL），两套并行，职责冲突 |
| `app/crawlers/` | ✅ 基本合理 | 分层清晰（base → 具体爬虫），但 async_base 状态不明 |
| `app/services/` | ✅ 基本合理 | 4 个服务各司其职，规模适中 |
| `app/utils/` | ⚠️ **分散** | 9 个工具模块，部分应归属 services（如 tfidf_matcher、recommendation_service） |

### 1.3 代码组织问题

1. **`src/` 子项目污染**：存在独立的 TypeScript 项目（DELIVERY.md、README.md、config.yaml），与 Python 主系统无直接关联，占用顶层目录
2. **入口分散**：`main.py`、`harvest_main.py`、`ccgp_main.py`、`web_server.py` 四个入口，缺乏统一抽象
3. **archive/ 目录未清理**：包含旧版本爬虫代码和 audit hooks，长期积累无维护

## 二、技术债评估

### 2.1 代码重复率

**高风险重复：**

- **两套数据库 API**：`app/database/db.py`（同步 SQLite）和 `app/database/async_models.py`（异步 PostgreSQL）完全并行，未统一
- **`routes_n8n.py` vs `routes_users.py`**：用户管理逻辑可能重复
- **爬虫基类重复**：`crawlers/base.py`（291 LOC）和 `crawlers/async_base/__init__.py`（385 LOC）职责重叠
- **TF-IDF 实现重复**：`app/utils/tfidf_matcher.py`（238 LOC）已存在于 services 层，同时 `qualification_matcher.py` 也有类似文本匹配逻辑

### 2.2 依赖复杂度

**直接依赖：16 个**（合理范围）

| 依赖 | 用途 | 备注 |
|------|------|------|
| fastapi + uvicorn | Web 框架 | ✅ 主流选择 |
| playwright + playwright-stealth | 爬虫 | ⚠️ 重型依赖，反爬功能可内联 |
| asyncpg + redis | 异步 DB/缓存 | ❌ asyncpg 已引入但未实际使用 |
| pandas + openpyxl | 数据处理 | ⚠️ 对采集系统而言偏重 |
| apscheduler | 定时任务 | ⚠️ FastAPI 生态已有 APSchedulerMiddleware |
| numpy | 数值计算 | ⚠️ 仅用于 anti_detect 随机指纹生成，可替代 |
| loguru | 日志 | ✅ 优于标准 logging |
| pydantic + pydantic-settings | 配置 | ✅ 类型安全 |

**问题：**
- `numpy` 仅用于 anti_detect 中的随机数生成（正态分布），可移除或替换为 Python 标准库 `random`
- `pandas` + `openpyxl` 用于 Excel 生成，对简单数据导出过于重型

### 2.3 配置管理

| 配置方式 | 文件 | 评估 |
|---------|------|------|
| `.env` | 环境变量 | ⚠️ 生产配置存在但分散 |
| `config/settings.py` | Pydantic BaseSettings | ✅ 统一管理 |
| `config/settings.json` | JSON 配置 | ❌ 冗余，未被代码引用？ |
| `config/admin_users.json` | 用户配置 | ⚠️ 明文 JSON，密码哈希在代码层 |
| `pyproject.toml` | 项目元数据 | ✅ 现代 Python 项目规范 |

**问题：**
- 三套配置格式并行（`.env`、`.py`、`.json`），增加维护复杂度
- `admin_users.json` 管理后台用户明文存放，不安全

### 2.4 错误处理一致性

| 模块 | 错误处理方式 | 一致性 |
|------|------------|--------|
| `routes/` | 部分 try/except pass，部分无异常处理 | ❌ 不一致 |
| `db.py` | 吞异常 + logger.error，事务 rollback | ⚠️ 安全但掩盖错误 |
| `crawlers/` | loguru 记录，无结构化异常 | ⚠️ 部分异常导致静默失败 |
| `harvest/exception_handler.py` | **状态机模式**（645 LOC）| ✅ 最完善，但过重 |

**关键问题：** `exception_handler.py` 实现了完整的状态机异常处理系统（Phase 1 标注），但大部分 routes 和 crawlers 完全未使用这套机制。

## 三、架构评估

### 3.1 分层清晰度

```
┌─────────────────────────────────────────────────┐
│  API Layer (routes/)          [20+ 路由模块]    │  ← 路由过碎
├─────────────────────────────────────────────────┤
│  Services Layer               [4 个服务]       │  ✅ 清晰
├─────────────────────────────────────────────────┤
│  Business Core (harvest/)      [6 模块 ~4K LOC] │  ⚠️ 膨胀
├─────────────────────────────────────────────────┤
│  Data Access (database/)      [db.py + async]  │  ❌ 冲突
├─────────────────────────────────────────────────┤
│  Infrastructure (crawlers/    [爬虫 + 工具]    │  ⚠️ 分散
│           utils/, middleware/)                  │
└─────────────────────────────────────────────────┘
```

**典型问题：**
- `routes_n8n.py` 直接调用爬虫逻辑，绕过了 services 层
- API 层直接操作 SQLite 连接（`_get_conn()`），违反分层原则

### 3.2 耦合度分析

**高耦合点：**

1. **`routes/projects.py` 直接引用 `SYS_PATH = Path(...)`** — 全局路径硬编码
2. **`db.py` 单例模式 + threading.local** — 难以测试，生命周期复杂
3. **`harvest/anti_detect.py`** — 1476 LOC 超大单一模块，包含指纹、WebGL、Canvas、TLS、DNS 随机化，违反单一职责
4. **`harvest/predictive_anomaly.py`** — 734 LOC，ML 异常检测系统紧耦合主流程

### 3.3 可扩展性

| 维度 | 评分 | 说明 |
|------|------|------|
| 新增爬虫 | ⚠️ 3/5 | 需要继承 base.py，但异步基础架构状态不明 |
| 新增 API 路由 | ⚠️ 3/5 | 路由文件过多，模板不统一 |
| 新增数据源 | ❌ 2/5 | 两套 DB 冲突，新增数据源需同时修改两套 |
| 新增功能模块 | ⚠️ 3/5 | harvest/ 模块已过重，新增功能没有明确位置 |

## 五、性能评估

### 5.1 数据库查询效率

| 问题 | 位置 | 影响 |
|------|------|------|
| `db.py` 无连接池 | 所有 DB 操作 | 高并发下性能差 |
| SQLite WAL 模式 | `db.py` | ✅ 已启用，适合读多写少 |
| N+1 查询 | `routes/projects.py` | ⚠️ 已用 `_batch_load` 优化但仅限 projects |
| 无查询索引覆盖 | 部分表 | ⚠️ `duplicate_records`、`data_cache` 可能全表扫描 |
| 缓存 TTL | `routes/projects.py` | ✅ 60s 内存缓存 |

**数据库架构问题（严重）：**
- `async_models.py`（672 LOC）定义了完整的 PostgreSQL asyncpg 架构，但**从未被实际使用**
- 项目实际运行在 SQLite 上，而 asyncpg 只是死代码

### 5.2 缓存策略

| 缓存层 | 实现 | 评估 |

| 爬虫缓存 | `harvest/cache_manager.py`（590 LOC）| ✅ 完整实现 |
| TF-IDF 缓存 | `routes/projects.py` | ⚠️ 5min TTL 但无 LRU 淘汰 |

### 5.3 异步处理

| 问题 | 说明 |
|------|------|
| `main.py` 使用 `asyncio.run()` | 单次执行，进程退出即结束 |
| `harvest_main.py` 异步入口 | ✅ 正确的 asyncio 模式 |
| `ccgp_main.py` 独立同步脚本 | ⚠️ 与 harvest_main.py 职责重叠 |
| `async_base/__init__.py`（385 LOC）| ❌ **未被任何爬虫导入使用** |
| `asyncpg` 引入但未使用 | 死代码，增加依赖体积 |
