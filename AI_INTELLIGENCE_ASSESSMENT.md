### 1.1 技术栈

| 任务调度 | APScheduler + 自定义 ConcurrencyScheduler | ✅ 已实现 |

### 1.2 数据流架构

```
爬虫采集（ccgp / cqggzy）
    ↓
关键词过滤（TenderFilter / TF-IDF 匹配器）
    ↓
详情页批量抓取（ConcurrencyScheduler 并发控制）
    ↓
数据标准化（TenderInfo → 22 字段字典）
    ↓
报表生成（Excel / JSON）+ Web API / n8n 推送
```

### 1.3 现有"类 AI"模块

| 模块 | 技术 | 评价 |
|------|------|------|
| `tfidf_matcher.py` | TF-IDF + jieba 分词 + 同义词扩展 | ⚠️ 基础，关键词严格匹配，无法捕捉语义 |
| `TenderFilter` | 规则关键词匹配 + 排除词 | ⚠️ 规则僵化，无法增量学习 |
| `ConcurrencyScheduler` | SafetyLevel 分级并发控制 | ✅ 架构良好，与 AI 任务调度可复用 |
| `TenderInfo` 模型 | 25 字段结构化数据 | ✅ 为 AI 特征提取奠定基础 |

### 1.4 核心痛点

1. **关键词匹配精度低** — 同义词/语义变体导致漏采和误采
2. **无自动分类** — 业务类型（政府采购/工程建设）、信息类型靠人工预设
3. **去重依赖 URL** — 相似项目不同链接会重复采集
4. **无优先级机制** — 所有匹配项目平铺，无法聚焦高价值标的
5. **无预测性告警** — 截标日期临近时无主动提醒
6. **数据沉睡** — 历史数据无法被语义检索

## 二、可 AI 化场景

### 场景 1：语义智能筛选（高优先级）

**问题**：现有 `TenderFilter` 仅支持关键词字面匹配，"智慧城市" 主题的标的若标题不含 "智慧" 关键词则全部漏采。

**解决方案**：用 Embedding 模型（如 `text-embedding-3-small` / 本地 `m3e`）对项目标题和摘要做向量检索，基于语义相似度而非字符串匹配过滤。

**预期效果**：召回率提升 30-50%，误采率降低 20%。

---

### 场景 2：自动分类与打标（中优先级）

**问题**：`business_type`（业务类型）和 `info_type`（信息类型）靠爬虫 Hardcode，无法自动学习新类别。

**解决方案**：训练/微调轻量中文文本分类模型（如 `BERT-chinese` / `Chinese-BERT-wwm`），对标题+摘要进行多标签分类。

**预期效果**：分类准确率 > 90%，减少人工归类工作量。

---

### 场景 3：智能去重与相似度检测（高优先级）

**问题**：现有去重仅按 URL 判断，同一项目不同页面版本会重复。

**解决方案**：用 Embedding 提取标题/摘要向量，计算余弦相似度，相似度 > 0.85 的合并或标记。

**预期效果**：重复率从当前 ~15% 降至 < 3%。

---

### 场景 4：异常检测与告警（中优先级）

**问题**：采集系统无异常感知能力，网站结构变化、字段缺失、采集量骤降无告警。

**解决方案**：基于历史数据（每日采集量、字段完整率、采集耗时）建立统计模型（Isolation Forest / Z-score），触发阈值自动告警。

**预期效果**：故障发现时间从小时级降至分钟级。

---

### 场景 5：项目优先级排序（机会型）

**问题**：所有匹配项目平铺，无法快速识别高价值标的。

**解决方案**：综合评分模型——输入：预算金额（结构化）、关键词匹配度（语义）、截标剩余天数、地区，输出：优先级分数（0-100）。

**预期效果**：Top 10 高价值标的优先推送。

---

### 场景 6：Deadline 临近提醒（低优先级）

**问题**：投标截止日期临近时无主动通知。

**解决方案**：定时任务扫描数据库，截标日期 ≤ 3 天的项目通过 n8n → 飞书/钉钉推送。

**预期效果**：降低因错过截止日期导致的废标风险。

---

### 场景 7：RAG 知识库问答（长期价值）

**问题**：用户想以自然语言查询历史数据（"近三月有哪些 AI 相关项目？"）。

**解决方案**：将历史结构化数据导入 RAGFlow，Embedding 存储到向量数据库，语义检索 + LLM 总结。

**预期效果**：对话式数据探索，替代手工 SQL 查询。

## 三、技术选型建议

### 3.1 向量 Embedding

| 方案 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| **本地 m3e-base**（`moka-ai/m3e-base`） | 免费、离线、中文优化 | 需要 GPU/CPU 资源 | ⭐⭐⭐⭐ |

## 六、架构演进建议

```
当前架构：
  Crawler → Filter(TF-IDF) → Store → Report

AI 化后架构：
  Crawler
    ↓
  Filter(TF-IDF 预过滤 + Embedding 语义精筛)
    ↓
  [AI Layer]
    ├─ Embedding Service（向量生成）
    ├─ Classifier（自动分类）
    ├─ Deduplicator（向量去重）
    ├─ Priority Scorer（评分排序）
    └─ Anomaly Detector（异常检测）
    ↓
  Store（PG + pgvector）
    ↓
  [Output]
    ├─ Report（Excel/JSON）
    ├─ n8n Push（飞书告警）
    └─ RAG Query（历史语义问答）
```

## 七、风险与注意事项

1. **API 费用**：OpenAI Embedding 按量计费，建议优先用已配置的 Nvidia NIM 或本地 m3e
2. **PG 迁移**：当前生产使用 SQLite，生产部署需迁移到 PostgreSQL + pgvector
3. **标注数据**：分类模型微调需少量人工标注（每个类别 50-100 条）
4. **延迟**：Embedding 推理有延迟，采集流水线需做异步批处理
5. **冷启动**：初期无历史数据时 Embedding 效果有限，建议 TF-IDF 降级兜底
